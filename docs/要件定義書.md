## 1. プロジェクト概要

### 1-1. 背景・課題

学習者 (hideharu-AI) はスクールで Web アプリ開発を学んでおり、これまで
**タスク管理アプリ (task-board)** → **レシピ管理アプリ (recipe-board)** → **SNS 風アプリ (sns-board)** と段階的に題材を変えてフルスタック開発を経験してきた。

4 つ目の題材として、**旅行の記録を時系列で残し、複数ユーザーが互いの旅行記録に対してコメント・いいねで反応できる「旅行記録アプリ (trip-diary)」** を選定。
1 ユーザー1記録の閉じた CRUD ではなく、**マルチユーザー双方向インタラクション**を扱う点が SNS-board からの継承テーマとなる。

### 1-2. 目的 (学習到達点)

本プロジェクトの学習姿勢は「**習う → 慣れる → マスター**」の3段階で進める。

| 段階 | 姿勢 | 目安 |
|---|---|---|
| 第 1 段階 (習う) | 公式チュートリアル・参考プロジェクト (recipe-board / sns-board) を写経・模倣しながら最低限の動作原理を理解する | Phase 1 立ち上げ初期 |
| 第 2 段階 (慣れる) | 完璧を求めず手を動かし、一連の流れ (DB → API → UI → デプロイ) を体感する。動くものを作ることを最優先 | Phase 1 〜 Phase 2 |
| 第 3 段階 (マスター) | 各機能の設計判断・トレードオフを言語化し、他人にレビューできる水準まで深掘る | Phase 3 〜完了後 |

> **「習うより慣れろ」段階では完璧主義で手が止まることを避ける。**
> 「マスター」段階では「なぜこの設計か」「他の選択肢はあったか」「どこに事故の芽があるか」を自分の言葉で説明できることを目標とする。

詳細は [学習ロードマップ.md](./学習ロードマップ.md) を参照。

#### 学習対象 (到達点の具体)

- マルチユーザー前提のドメイン設計を経験する (他者のコメント・いいねを想定)
- 1 対多 (Trip → DayEntry) のネスト構造を扱う (recipe-board の「材料・手順」を発展)
- 認証 (JWT in HttpOnly Cookie) と認可 (自分の記録のみ編集・削除可) を実装する
- 画像アップロード (Phase1: ローカルディスク / Phase3: S3) を扱う
- 非正規化カウンタ (いいね数 / コメント数) の同期の難しさを学ぶ
- AWS 無料枠での本番デプロイを経験する (Phase3)

### 1-3. 対象ユーザー

- 学習用アプリだが **マルチユーザー前提**で設計
- 想定利用者数: 5〜20 名 (同時利用 5 名程度)
- ユーザー属性: 知人 / スクール仲間 / 講師 (招待制を想定、誰でも登録できる仕様にはしない)

### 1-4. 利用環境

| 区分 | 環境 |
|---|---|
| ブラウザ | Chrome / Safari 最新版 |
| 解像度 | PC (1280px〜) を主に想定。タブレット (768px〜) は最低限崩れない |
| デバイス | PC・タブレット (スマートフォン対応は Phase3 で検討) |

### 1-5. 提出時の完成スコープ

本プロジェクトは Phase 1 〜 Phase 3 を順次実装する計画だが、**最終的にはスクール提出までに講師指定の全機能・全画面を完成させること**を最終ゴールとする。
Phase 分割は学習進度を整理するための便宜であり、提出物としての受け入れ条件ではない。
講師指定要件と Phase の対応は §10「講師要件マッピング表」を参照。

## 2. 用語定義

| 用語 | 定義 |
|---|---|
| 旅行記録 (Trip) | 1 回の旅行を表すルートエンティティ。タイトル・期間・場所・本文を持つ |
| 出来事 (DayEntry) | 旅行中の 1 日 (または 1 シーン) のメモ。Trip に複数ぶら下がる (recipe-board の「材料・手順」相当のネスト) |
| 公開範囲 | 旅行記録の閲覧可否制御。`public` / `friends` / `private` (Phase2 で導入) |
| いいね (Like) | 旅行記録に対する 1 ユーザー 1 回の反応 |
| コメント (Comment) | 旅行記録に対するテキスト反応 (140 文字以内) |
| 投稿者 | 旅行記録を作成したユーザー (User) |

## 3. 機能要件 (サマリ)

詳細は [機能一覧.md](./機能一覧.md) を参照。本セクションは Phase 別サマリのみ。

### 3-1. Phase 1 (MVP) — 本セッションで実装

| カテゴリ | 機能 |
|---|---|
| 認証 | サインアップ / ログイン / ログアウト (JWT in HttpOnly Cookie) |
| 旅行記録 | 一覧 / 詳細 / 作成 / 編集 / 削除 (削除は自分のもののみ) |
| 出来事 (ネスト) | 旅行記録に紐づく日別メモを 1 画面で複数登録・編集 |
| コメント | 投稿 / 削除 (自分のもののみ) / 一覧表示 |
| いいね | 追加 / 取り消し / 件数表示 |
| 画像 | 旅行記録に複数画像添付 (ActiveStorage / ローカルディスク) |

### 3-2. Phase 2 (UX 拡張 + 交流 + プランニング)

> **方針**: 「凝ったアプリ」を目指し、Phase 2 で**マルチユーザー交流**と**旅行プランニング/予算管理**まで踏み込む。

| カテゴリ | 機能 |
|---|---|
| 公開範囲 | 旅行記録に `public` / `friends` / `private` を設定 |
| プロフィール | 表示名・自己紹介・アバター編集 |
| いいねしたユーザー一覧 | モーダル表示 |
| タグ | 自由記述タグ + タグ別タイムライン + 人気タグクラウド |
| カテゴリ | 国内 / 海外 / 一人旅 / グルメ / 世界遺産 / 家族旅 / アウトドア / 出張 等から主カテゴリ選択 |
| 検索・ソート | タイトル / 場所 / タグ / カテゴリ / 期間 の高度検索 (AND 複合可) + 新着・人気・タイトル順 |
| フォロー | 他ユーザーをフォロー / アンフォロー / 一覧 / フォロー中タイムライン (タブ切替) |
| **旅行プランニング** | 旅行作成時に「計画モード」で予定スポットを to-do 形式で登録 → 実施済チェックで DayEntry に昇格 / 進捗バー |
| **予算追跡** | カテゴリ別予算入力 (交通 / 宿 / 食事 / 観光 / お土産) + 各 DayEntry に支出記録 + Chart.js で予算-実績グラフ |

### 3-3. Phase 3 (本番デプロイ + 視覚化)

| カテゴリ | 機能 |
|---|---|
| **地図ピン表示** | タイムライン全 trip を Leaflet / MapLibre で地図表示 |
| **位置情報入力** | DayEntry / planned_spot に緯度経度を Nominatim 等で検索入力 |
| **ルート可視化** | 旅行詳細で DayEntry を時系列で線描画 |
| 通知センター | コメント / いいね / フォロー受信通知 (verb 型) + 既読管理 + 未読バッジ |
| **PDF エクスポート** | 旅行記録の A4 PDF 化 (画像込み) + 予算分析グラフ PDF (Prawn or wicked_pdf) |
| S3 移行 | 画像保存先を ActiveStorage Disk → S3 へ |
| AWS デプロイ | EC2 (t3.micro) + RDS (db.t3.micro) + EIP + Nginx (Terraform 構成) |
| 監視 | CloudWatch Logs / アラーム (ログ・監視・障害対応設計書 §3-3 と連動) |

### 3-4. Phase 4 (発展余地・任意)

| カテゴリ | 機能 |
|---|---|
| **DM (1:1)** | 相互フォロー間のダイレクトメッセージ / スレッド表示 / 既読管理 |
| リアルタイム | ActionCable で いいね / コメント / DM の即時反映 |
| プッシュ通知 | Web Push API + Service Worker でブラウザ通知 |
| PWA | オフライン閲覧 / インストール対応 (Workbox) |
| 多言語 | i18n (ja / en) |

## 4. 非機能要件

### 4-1. 性能

| 項目 | 目標値 |
|---|---|
| 一覧画面 初回表示 | 2 秒以内 (ローカル開発環境) |
| 詳細画面 表示 | 1 秒以内 |
| 同時接続数 | 5〜10 ユーザー (学習用想定) |

### 4-2. セキュリティ

- パスワードは BCrypt でハッシュ化 (Rails `has_secure_password`)
- 認証は JWT in **HttpOnly + Secure (本番) + SameSite=Lax** Cookie で保持 (XSS による窃取耐性)
- CSRF: SPA + JWT Cookie 構成のため、`SameSite=Lax` で基本対策 + 状態変更系 API には独自ヘッダ (`X-CSRF-Token` 等) を Phase 3 で検討
- UGC (本文・コメント) は Rails 側でサニタイズ (HTML タグはエスケープ表示)
- 画像アップロードは MIME / 拡張子 / サイズ (5MB) 制限
- 機密情報 (`.env` / SECRET_KEY_BASE / JWT_SECRET) は **絶対にコミットしない**

### 4-3. 可用性・運用

- Phase 1〜2 はローカル開発のみ
- Phase 3 で AWS デプロイ。バックアップは RDS 自動スナップショット (7日間保持)
- 障害対応は recipe-board の OPERATIONS.md を踏襲して整備予定

### 4-4. 拡張性

- API は `/api/v1/` で名前空間を切り、将来的な v2 移行を想定
- フロントエンドは Nuxt の Pinia ストアで状態管理し、画面追加を容易にする
- 画像保存先は ActiveStorage で抽象化し、Disk ↔ S3 の切替を 1 設定で行えるようにする

### 4-5. アクセシビリティ

- フォームには `<label>` を必ず付与
- ボタン・リンクは aria 属性を最低限付与
- 色だけに依存した情報表現を避ける (アイコン併用)

| レイヤー | 採用技術 | 採用理由 |
|---|---|---|

| 画像 | ActiveStorage (Disk / Phase3 で S3) | Rails 標準で実装容易 |
| CSS | Tailwind CSS | recipe-board と同構成 |
| インフラ (Phase3) | AWS EC2 + RDS + Terraform | recipe-board のパターンを流用 |

## 7. データモデル (サマリ)

詳細は [ER図.md](./ER図.md) を参照。

主要エンティティ:

- `users` — ユーザー (email / display_name / password_digest / bio / avatar)
- `trips` — 旅行記録 (user_id / title / destination / started_on / ended_on / body / visibility / likes_count / comments_count)
- `day_entries` — 出来事 (trip_id / day_number / happened_on / title / body / position)
- `comments` — コメント (trip_id / user_id / body)
- `likes` — いいね (trip_id / user_id) ※複合ユニーク
- `active_storage_*` — Rails 標準 (画像保存)
- (Phase2) `tags`, `trip_tags`, `follows`
- (Phase3) `notifications`

## 9. リスク・前提

| # | リスク / 前提 | 対策 |
|---|---|---|
| R-01 | TypeScript 不採用のため大規模化で型不整合バグが増える可能性 | Phase4 以降で TS 化を検討。Phase1〜3 は規模を抑える |
| R-02 | 画像アップロードを Disk で行うと EC2 容量を圧迫する | Phase3 で S3 移行 |
| R-03 | コメント・いいねの非正規化カウンタが本数とズレる可能性 | counter_cache 採用 + 月次バッチで再計算 (Phase3) |
| R-04 | マルチユーザー前提だが招待制のため新規登録 UI の安全性は中優先度 | Phase2 で reCAPTCHA / メール確認を検討 |
| R-05 | 本セッションで MVP まで実装するため Phase 後半の認可漏れ検証が薄い | Phase 2 冒頭で認可テスト (`security-audit-*.md`) を整備 |
