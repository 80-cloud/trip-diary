## 0. 本書の位置づけ

本書は trip-diary における **観測可能性 (Observability)** の設計指針をまとめる。
「障害を未然に検知 → 再現 → 修正 → 再発防止」のサイクルを学習教材として体系化することを目的とする。

> **学習姿勢への補足:** 「習う→慣れる→マスター」 ([学習ロードマップ.md](./学習ロードマップ.md) §1) の文脈で、本書の最大の目的は **「観測する習慣を身につける」** ことにある。
> 高機能な監視ツールを入れること自体が目的ではなく、「サービスが今どう動いているかを言語化できる」状態に到達することが「マスター」段階の到達点である。

## §2 構造化ログ設計

### §2-1 ログレベル方針

**対象 Phase: Phase 1 (現状から実施可能)**

| レベル | 用途 | 例 |
|---|---|---|
| `fatal` | プロセス継続不可 | DB 接続恒久断 / 認証鍵欠落 |
| `error` | 即対応必要 | 未処理例外 / 500 系応答 / S3 アップロード失敗 (Phase 3) |
| `warn` | 注視 / 集計対象 | リトライ / 認証失敗連続 / 遅いクエリ |
| `info` | 正常系の重要イベント | アプリ起動 / ユーザー登録 / 投稿作成 |
| `debug` | 開発時のみ | SQL / リクエスト内容 (本番出力禁止) |

- 環境変数 `RAILS_LOG_LEVEL` で切替 (既存 `production.rb` で対応済)
- 本番 (Phase 3): `info` 想定
- 開発: `debug`

### §2-2 JSON 構造化フィールド設計

**対象 Phase: Phase 3 (lograge 導入時)**

| フィールド | 型 | 説明 |
|---|---|---|
| `timestamp` | ISO8601 UTC | ログ生成時刻 |
| `level` | string | error / warn / info / debug |
| `logger` | string | Rails / ActiveRecord / ActionController 等 |
| `service` | string | "trip-diary-api" 固定 |
| `env` | string | production / staging / development |
| `request_id` | string | Rails 標準 X-Request-Id |
| `user_id` | int? | 認証済みリクエストのみ (CurrentAttributes 経由) |
| `method` | string | GET / POST / ... |
| `path` | string | "/api/v1/trips" |
| `status` | int | HTTP ステータス |
| `duration_ms` | float | レスポンスタイム |
| `error_class` | string? | 例外発生時のクラス名 |
| **`trip_id`** | int? | trip-diary 固有: 操作対象 trip |
| **`image_count`** | int? | trip-diary 固有: 添付画像数 |

> **学習補足 (マスター段階):** sns-board の JSON フィールド (Spring Boot + Logback) と本表を比較し、フレームワーク間の対応関係を理解する。詳細は §2-4 マッピング表参照。

### §2-3 PII (個人情報) マスキング方針

**対象 Phase: Phase 1 で着手 (本 PR で `config/initializers/filter_parameter_logging.rb` を整備)**

#### 禁止: 生ログ出力してはいけないもの

- パスワード / `password_confirmation`
- JWT トークン (`Authorization` ヘッダ / `trip_diary_token` Cookie)
- メールアドレス (必要なら user_id で代替)
- パスワードリセットトークン (Phase 2 以降)
- 画像バイナリ本体 (ファイル名のみ可)

#### 実装

`backend/config/initializers/filter_parameter_logging.rb` に、Rails 8 デフォルトの
`:passw, :email, :secret, :token, :_key, :crypt, :salt, :certificate, :otp, :ssn, :cvv, :cvc` に加えて、
trip-diary では以下を追加:

```ruby
Rails.application.config.filter_parameters += [
  :jwt, :authorization, :api_key
]
```

#### `filter_parameters` の効果範囲 (重要な注意点)

`config.filter_parameters` は **リクエスト「パラメータ」** (query string / form body / JSON body) を対象とする。
**HTTP ヘッダ (Authorization / Cookie 等) はフィルタ対象外**。

- ✅ Rails のデフォルトログには HTTP ヘッダは出力されないため、**通常運用では問題ない**
- ⚠️ 意図的に `request.headers["Authorization"]` をログ出力する場合は別途対応が必要
- 📋 **Phase 4 ⏸**: lograge 導入 + custom_options で Authorization / Cookie マスキング middleware (現状 Rails default ログで Authorization は非出力のため致命的問題なし / Phase 4 で必要時に追加)

### §2-4 リクエスト ID とコンテキスト伝播

**対象 Phase: Phase 1〜3 (段階拡張)**

#### Phase 1 (現状)

- Rails 標準の `X-Request-Id` ヘッダを受信し、`log_tags = [:request_id]` で出力 (既存 `production.rb`)

#### Phase 3 拡張

- `ActiveSupport::CurrentAttributes` でリクエスト中の `current_user.id` を保持
- lograge の custom_options で `user_id` をタグ出力

#### sns-board (Spring Boot MDC) との対応

| sns-board (Logback MDC) | trip-diary (Rails) |
|---|---|
| `MDC.put("traceId", ...)` | `Rails.logger.tagged(request_id)` |
| `MDC.put("userId", user.id)` | `Current.user_id = current_user.id` (Phase 3) |
| `@Slf4j` でログ取得 | `Rails.logger.info(...)` |
| logstash-logback-encoder | `lograge` gem |

### §2-5 出力先と保管方針

| Phase | 出力先 | 保管期間 |
|---|---|---|
| Phase 1 | STDOUT (Foreman/Puma 経由でターミナル表示) | なし (シェルバッファのみ) |
| Phase 2 | `backend/log/development.log` (Rails 標準) | ローカルファイル |
| Phase 3 | STDOUT → **CloudWatch Logs Agent** → CloudWatch Logs ロググループ `/trip-diary/rails` `/trip-diary/nginx` | **30 日保持** (AWS 無料枠内) |

### §2-6 フロントエンド (Nuxt) のログ方針

| Phase | 方針 |
|---|---|
| Phase 1 | `console.error` のみ (ブラウザ DevTools で目視) |
| Phase 3 | `$fetch.onResponseError` で集約 → 上位コンポーネントに通知 |
| Phase 4 (任意) | Sentry SDK 導入 → ブラウザ例外を集約 |

> **学習補足:** 「習う」段階では console を読む癖をつけることが最優先。Sentry のような外部サービスは「マスター」段階で導入意義を判断する。

## §3 運用・監視設計

### §3-1 SLI / SLO 設計

**対象 Phase: Phase 1 は目標のみ / Phase 3 で実測**

| SLI (指標) | SLO (目標) | 計測方法 (Phase 3) |
|---|---|---|
| ログイン成功率 | ≥ 99% | `/api/v1/login` の 2xx / 全リクエスト |
| Trip 詳細 API レイテンシ p95 | < 500 ms | アクセスログから集計 |
| 画像アップロード成功率 | ≥ 95% | `POST /api/v1/trips` (with images) の 2xx 率 |
| 月間可用性 | ≥ 99% | CloudWatch Synthetics or `/up` ヘルスチェック |

SLO 違反時の対応は §4 障害対応フローへ。

### §3-2 4 ゴールデンシグナル

**対象 Phase: Phase 3**

| シグナル | 監視対象 | CloudWatch メトリクス例 |
|---|---|---|
| **Latency** | エンドポイント別 p50 / p95 / p99 | カスタムメトリクス (lograge → CloudWatch Logs Insights) |
| **Traffic** | RPS / エンドポイント別 | アクセスログから集計 |
| **Errors** | 5xx 発生数・率 / 未処理例外 | CloudWatch Logs Insights |
| **Saturation** | CPU / Memory / Disk / DB 接続数 | EC2: `CPUUtilization` `MemoryUtilization` (CW Agent) / RDS: `DatabaseConnections` `FreeStorageSpace` |

### §3-3 trip-diary 固有の監視項目

**対象 Phase: Phase 3 (Phase 1〜2 で `warn` ログだけ仕込んでおくのは可)**

マルチユーザー双方向アプリ特有として:

| 機能 | 監視項目 | 閾値 (warn / critical) | 対応 |
|---|---|---|---|
| **F-AUTH 認証** | 同一 IP からの `/api/v1/login` 401 回数 | 1 分 10 回 / 1 分 100 回 | ブルートフォース検知 → R-3 |
| **F-AUTH JWT** | 不正署名 / 期限切れ token 検出数 | 1 分 5 件 / 1 分 50 件 | 鍵漏洩 or 盗難疑い → R-3 |
| **F-CMT コメント** | 同一ユーザーの 1 分あたり投稿数 | 1 分 20 件 / 1 分 100 件 | スパム / Phase 4 でレート制限 |
| **F-LIKE いいね** | 同一ユーザーの 1 分あたり toggle 回数 | 1 分 100 回 / 1 分 500 回 | 連打スパム |
| **F-IMG 画像** | 5MB 超過 / 6 枚目以降の弾き発生数 | 集計のみ (アラートなし) | UX 改善余地の判断材料 |
| **F-IMG S3** | アップロード失敗数 (Phase 3) | 5 分 5 件 / 5 分 30 件 | IAM / S3 到達性 → R-4 |

### §3-4 ヘルスチェック

**対象 Phase: Phase 1〜3**

- 既存 `/up` (Rails 8 標準) を継続使用
- `production.rb` で `silence_healthcheck_path = "/up"` 設定済 (ログ抑制)
- Phase 3 で Nginx → `/up` を 30 秒間隔で監視 + 3 回連続失敗で Puma 再起動 (systemd `Restart=on-failure`)
- ヘルスチェックエンドポイント自前版 `/api/v1/health` も併存 (現状: `Api::V1::HealthController#show` で `{status: "ok", time: ...}` を返す)

### §3-5 ログ集約・可視化ツール選定

**対象 Phase: Phase 3**

| 候補 | 採用判断 | 理由 |
|---|---|---|
| **CloudWatch Logs** | ✅ 第一選択 | AWS 無料枠内 (5GB/月) で完結 |
| **CloudWatch Logs Insights** | ✅ クエリ用 | SQL 風クエリで集計可能 |
| **Datadog 無料枠** | 🔄 Phase 4 検討 | 強力だがホスト数制限あり / 学習負荷増 |
| **Prometheus + Grafana** | ❌ Phase 4 以降 | 学習目的なら最終段階 |
| **Sentry (エラー集約)** | 🔄 Phase 3 後半オプション | 例外集約と通知に強い |

## §4 障害対応フロー

### §4-1 重大度 (Severity) 定義

**対象 Phase: Phase 1 で方針確定**

| Sev | 定義 | 対応速度 | 通知 (Phase 3) |
|---|---|---|---|
| **Sev1 (緊急)** | サービス全断 / DB 接続不可 / 5xx > 50% / データ損失 | 即時 (15 分以内に着手) | メール + Slack 直接 |
| **Sev2 (高)** | 特定機能不可 (Trip 作成 5xx / 画像 upload 全件失敗) / 5xx > 10% | 1 時間以内 | メール |
| **Sev3 (中)** | 一部劣化 (レイテンシ p95 > 2s / 散発エラー) | 翌営業日 | 記録のみ |
| **Sev4 (低)** | ログ警告のみ / ユーザー影響なし | 都合の良い時 | 記録のみ |

### §4-2 検知 → 対応 → 復旧フロー

**対象 Phase: Phase 3 (本格運用) / Phase 1 は簡易フローで代替**

#### Phase 3 想定フロー

```
① CloudWatch アラーム発火
        ↓
② メール受信 → トリアージ (request_id で影響範囲確認)
        ↓
③ 一次対応 (止血優先 / Runbook 参照)
        ↓
④ 必要ならエスカレーション (今は単独運用)
        ↓
⑤ 復旧確認 (メトリクスが正常範囲に戻ったか)
        ↓
⑥ ポストモーテム (5 Whys / 再発防止策 → Issue 起票)
```

#### Phase 1 (現状) 簡易フロー

```
① ローカルでエラー再現
        ↓
② 原因特定 (Rails ログ / Chrome DevTools)
        ↓
③ 修正 + テスト追加 (回帰防止)
        ↓
④ PR → CI 緑 → マージ
        ↓
⑤ 学習ノート (docs/学習ロードマップ.md §5) に振り返り記入
```

### §4-3 アラート閾値表

**対象 Phase: Phase 3**

| メトリクス | Warning | Critical | Severity | Runbook |
|---|---|---|---|---|
| 5xx 応答率 | 5 分間 > 1% | 5 分間 > 5% | Sev2 | R-1 |
| API p95 レイテンシ | 5 分間 > 1s | 5 分間 > 3s | Sev3 → Sev2 | R-2 |
| `/up` ヘルスチェック失敗 | 1 回失敗 | 3 回連続失敗 | Sev2 → Sev1 | R-1 |
| EC2 CPU 使用率 | 5 分間 > 70% | 5 分間 > 90% | Sev3 | R-5 |
| RDS 接続数 | 上限の 70% | 上限の 90% | Sev2 | R-2 |
| RDS FreeStorageSpace | < 2 GB | < 500 MB | Sev2 → Sev1 | (容量逼迫 — 別 Runbook で詳細化) |
| 認証 401 連続 (同一 IP) | 1 分 10 回 | 1 分 100 回 | Sev3 → Sev2 | R-3 |
| 画像 upload 失敗数 | 5 分 5 件 | 5 分 30 件 | Sev3 → Sev2 | R-4 |

### §4-4 Runbook 一覧

**対象 Phase: Phase 1 で骨子 / Phase 3 で本格運用**

| ID | シナリオ | 対象 Phase | 一次対応概要 |
|---|---|---|---|
| **R-1** | Rails が起動しない | Phase 1〜3 | bundle install → `bin/rails zeitwerk:check` → Gemfile.lock の整合 → DB マイグレーション |
| **R-2** | DB 接続不可 | Phase 1〜3 | Phase 1: `docker compose ps` で MySQL 状態確認 / Phase 3: RDS コンソール + SG 確認 |
| **R-3** | 認証エラー多発 (401 急増 / JWT 異常) | Phase 1〜3 | `JWT_SECRET` 環境変数の一致確認 → Cookie ドメイン / SameSite 設定見直し → 攻撃元 IP 特定 (Phase 3) |
| **R-4** | 画像アップロード失敗 | Phase 1〜3 | ActiveStorage Disk: ディスク容量 / Phase 3 S3: IAM 権限 + バケット名 + リージョン |
| **R-5** | 5xx 急増 / レイテンシ悪化 | Phase 3 | request_id で代表ログ追跡 → N+1 / 遅いクエリ / DB 接続枯渇 / EC2 リソース確認 → ロールバック or 機能一時停止 |

各 Runbook の詳細手順は別ファイル `docs/runbooks/R-N.md` (Phase 2 以降) に分割予定。本書は **目次** として機能する。
