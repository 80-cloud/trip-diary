## 1. Phase 1 / 2: ローカル開発環境

### 1-1. プロセス構成

| プロセス | 起動方法 | ポート |
|---|---|---|
| MySQL 8 | `docker compose up -d db` | 3316 (host) → 3306 (container) |
| Rails API | `cd backend && bin/rails s -p 3010 -b 0.0.0.0` | 3010 |
| Nuxt | `cd frontend && PORT=3011 npm run dev` | 3011 |

### 1-3. 環境変数 (`.env`)

`.env.example` をコピーして `.env` を作成し、各種シークレットを設定する。**`.env` は絶対にコミットしない**。

```env
MYSQL_ROOT_PASSWORD=...
MYSQL_DATABASE=trip_diary_dev
MYSQL_USER=trip
MYSQL_PASSWORD=...
MYSQL_PORT=3316
RAILS_PORT=3010
SECRET_KEY_BASE=...        # bundle exec rails secret
JWT_SECRET=...             # openssl rand -hex 64
NUXT_PORT=3011
NUXT_PUBLIC_API_BASE=http://localhost:3010/api/v1
```

### 2-2. AWS リソース (Terraform 管理予定)

| リソース | スペック | 月額目安 (無料枠) | 備考 |
|---|---|---|---|
| EC2 | t3.micro × 1 | 無料 (12ヶ月) | Nginx + Rails + Nuxt 同居 |
| RDS | db.t3.micro / MySQL 8 / 20GB | 無料 (12ヶ月) | 自動 SS 7 日 |
| EIP | × 1 (EC2 付与) | 無料 (付与中のみ) | |
| S3 | 1 バケット / Standard | 無料枠内 (5GB) | 画像保存 |
| Route53 | (任意) | $0.5/月 | ドメイン取得時 |
| CloudWatch Logs | 5GB/月 | 無料枠内 | 簡易監視 |
| VPC / Subnet / SG / IGW | - | 無料 | 標準構成 |

### 2-3. デプロイフロー (案)

```
git push origin main
        │
        ▼
GitHub Actions
        │
        ├─→ Rails: bundle install → rails assets:precompile
        ├─→ Nuxt: npm ci → npm run build
        ├─→ scp/rsync → EC2
        └─→ ssh EC2 → systemctl restart puma / nginx -s reload
```

> Phase 3 では GitHub Actions を Issue ベースで段階導入。最初は手動 `scp` でデプロイし、慣れた段階で自動化する (学習ロードマップ §3 参照)。

### 2-4. セキュリティ・運用

| 項目 | 方針 |
|---|---|
| 通信 | 80 → 443 リダイレクト (Let's Encrypt) |
| DB アクセス | RDS は private subnet のみ。EC2 SG から 3306 のみ許可 |
| SSH | キーペア認証 / 22 は My IP のみ許可 |
| 機密情報 | EC2 上の `.env` で管理 (リポジトリには絶対に含めない) |
| バックアップ | RDS 自動スナップショット (7 日保持) |
| ログ | アプリログを CloudWatch Logs Agent で集約 (Phase3 後半) |
| 監視 | CloudWatch アラーム (CPU > 80% / 5xx 急増) |

---

## 3. コスト管理

- AWS 無料枠を超えないよう、t3.micro / db.t3.micro を選択
- 検証完了後は `terraform destroy` で全削除 (講師レビュー後)
- S3 はライフサイクルポリシーで 90 日以上経過した画像を Glacier へ (Phase3 後半)

---

## 4. 障害対応 (Phase3 で詳細化予定)

- 5xx 急増時: Nginx / Puma / Rails ログを CloudWatch で確認
- DB 接続不可: RDS コンソール / SG 設定確認
- ディスク容量逼迫: 画像 S3 化 / ログローテーション
- 復旧手順: `OPERATIONS.md` を Phase3 で整備 (recipe-board の体系を流用)
