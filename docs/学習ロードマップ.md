## 0. このドキュメントの目的

本プロジェクト (trip-diary) の学習目的は「動くものを作る」ことだけでなく、**「なぜこの設計を選んだか」を自分の言葉で説明できる水準に到達すること**にある。
そのための学習段階を「**習う → 慣れる → マスター**」の 3 段階に整理し、各 Phase でどの姿勢で取り組むかを明文化する。

---

## 1. 3 段階の定義

| 段階 | 姿勢 | 行動指針 | 完了の目安 |
|---|---|---|---|
| **習う** | お手本に従う | recipe-board / sns-board / Rails ガイド / Nuxt 公式チュートを写経 | 自分一人で `rails new` → `bin/rails s` → ブラウザに表示 が止まらずできる |
| **慣れる** | 動かす | 完璧主義で止まらない。バグはあとで直す。一連の流れ (DB→API→UI) を 1 周回す | Phase 1 MVP が動く |
| **マスター** | 言語化する | 「なぜこの設計か」「他の選択肢は何か」「事故の芽はどこか」を口頭/文書で説明 | docs に振り返り (learning-notes) を書き、PR にレビューを受ける |

---

## 2. Phase × 段階 マッピング

| Phase | 主な段階 | 重点 |
|---|---|---|
| Phase 1 (MVP) | 習う / 慣れる | 写経 + 一連の流れの体感。**完璧主義で止まらない** |
| Phase 2 (発展機能) | 慣れる / マスター | 設計判断を意識し始める。コードレビュー前に「なぜこう書いたか」を言語化 |
| Phase 3 (AWS デプロイ) | マスター | 本番固有の事故 (SG / IAM / ログ漏れ) を経験。Runbook (`OPERATIONS.md`) を書く |
| Phase 4 (発展余地) | マスター | リアルタイム / PWA / 多言語 — 各機能の「なぜ必要か」を判断 |

---

## 3. Phase 1 で守りたい姿勢 (本セッション)

1. **動くを最優先**。Lint / 型 / テストの完璧さは Phase 2 へ持ち越して良い
2. ただし**機密情報 (`.env` / SECRET) は最初から守る**。後から対応は事故の元
3. **写経した箇所には「参考: recipe-board の XX」と PR / 学習ノートにメモを残す** — マスター段階で振り返るため
4. つまずいたら **Claude Code に「概念を簡単な言葉で説明して」と聞く**。原理を 1 つずつ吸収

## 4. マスター段階で問う観点 (振り返りチェックリスト)

実装した各機能について以下の問いに答えられるか確認:

### 4-1. 認証 (JWT in HttpOnly Cookie)

- なぜ localStorage ではなく HttpOnly Cookie に保存したか？
- CSRF はどう対策しているか？SameSite=Lax の限界は？
- access + refresh の 2 段方式に拡張するとしたら、どこを変える？

### 4-2. 旅行記録 CRUD + DayEntry (ネスト)

- `accepts_nested_attributes_for` を使う利点 / 欠点は？
- フォームから子要素を 1 件削除するときの ID 渡し方は？
- N+1 をどこで防いだか？(`includes(:day_entries)` の入れ忘れ事故)

### 4-3. コメント・いいね (counter_cache)

- カウンタが実数とズレるシナリオは？(例: race condition)
- 1 人 1 記録 1 いいねを保証する仕組みは？(DB UNIQUE / アプリ層 validation の二重防衛)

### 4-4. 画像アップロード (ActiveStorage)

- Disk と S3 を切り替えるとき、どの設定を変えるだけで済むか？
- 5MB 制限はどこで効かせているか？(クライアント / Rails / Web サーバ の三層)

### 4-5. 認可 (自分のデータのみ編集削除)

- 認可漏れが発生しやすい場所は？(`update` / `destroy` のリソース取り違え)
- before_action と Pundit / CanCanCan の違いは？

### 4-6. ソーシャル交流 (Phase 2: フォロー / 通知)

- 自己フォロー禁止 / 相互フォロー判定の SQL は？
- 通知の verb 型設計 (commented / liked / followed) — STI vs verb カラム
- 既読フラグ vs 既読時刻の使い分け
- N+1 を避ける `includes(:actor, :target)` (polymorphic)

### 4-7. 旅行プランニング (Phase 2: 計画 → 実記録の状態遷移)

- 「計画モード」と「実記録モード」を 1 モデルで状態管理する vs 別モデルに分ける
- planned_spot を DayEntry に昇格する設計 (新規 record 作成 vs 同 row 状態変更)
- 進捗バーの計算 (counter_cache vs SUM クエリ)

### 4-8. 予算管理 + 可視化 (Phase 2: Chart.js)

- 予算 vs 実績の集計 SQL (GROUP BY category)
- Chart.js の Vue 統合 (vue-chartjs) と SSR の罠
- 通貨単位 / 小数桁の DB 表現 (decimal vs integer)

### 4-9. 地図・位置情報 (Phase 3: Leaflet / Nominatim)

- Leaflet のタイル選択と利用規約 (OpenStreetMap)
- polymorphic `locations` テーブルの設計 (locatable_type / locatable_id)
- Nominatim のレート制限とキャッシュ戦略

### 4-10. PDF エクスポート (Phase 3: Prawn / wicked_pdf)

- 日本語フォント埋め込み (IPA 等)
- 画像の埋め込みサイズと PDF ファイルサイズ
- グラフ画像を Chart.js から PNG 化して PDF に貼る方法

### 4-11. リアルタイム + DM (Phase 4: ActionCable)

- WebSocket 接続認証 (Cookie 経由) と認可
- broadcast の対象選定 (チャンネル設計)
- 再接続戦略 / メッセージ重複排除

### 4-12. PWA / 多言語 (Phase 4)

- Service Worker のキャッシュ戦略 (network-first / cache-first)
- i18n のキー設計と Vue I18n / Rails I18n の橋渡し

### 4-13. UX 強化 (Phase 2 / 機能候補第 2 弾)

- フォーム自動保存: `debounce` の窓 / `localStorage` キーの衝突回避 / 離脱時のリストア UX
- 下書き (status enum): scope (`Trip.published`) と認可 (下書きは投稿者のみ) の両立
- 無限スクロール: cursor pagination の cursor 設計 (id vs created_at + tiebreak)
- 遅延読込: IntersectionObserver / loading="lazy" / SSR との相性
- D&D 並び替え: position カラム再採番 vs gap 戦略 (insert 時の rebalance)
- 検索履歴: localStorage vs DB の使い分け (個人デバイス vs 同期)

### 4-14. 統計と可視化 (Phase 3 / F-STATS-*)

- オンデマンド集計 vs キャッシュ (solid_cache) の切替閾値
- 都道府県判定: 文字列マッチ vs 専用 `prefecture` カラム
- 日本地図 SVG: topojson vs 自前 SVG / クリック領域の hit-test
- 季節判定: 月ベース vs 気象学的 (3-5月=春) のどちらを採るか
- Chart.js のレスポンシブ対応とダークモード追従

### 4-15. グループ編集と競合解決 (Phase 4 / F-GROUP-*)

- 楽観ロック (`lock_version`) と悲観ロック (`SELECT FOR UPDATE`) の使い分け
- 競合検知時のマージ UI (どっちを優先するか / フィールドごとのマージ)
- 招待フロー (メール vs アプリ内通知)
- 監査ログ (誰がいつ何を変えたか) の設計 — Phase 5 で audit table 検討

### 4-16. 不採用候補から学ぶこと (機能一覧 §参考資料)

- なぜ「動作担保困難」と判断したか (外部 API レート制限 / ライセンス / モバイル App 領域)
- 将来再採用するための条件 (有料 API 採用許容 / プラットフォーム変更 / etc.)
- 「やらない判断」も設計の一部であることを言語化する練習
