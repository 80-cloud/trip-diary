## §3 trip-diary 実施結果

### E-H1: signup の field-level エラー漏洩 — 🔴 **要修正 (本 PR で fix)**

- **症状**: `backend/app/controllers/api/v1/auth_controller.rb:10` で `render json: { errors: user.errors.full_messages }` をそのまま返却 → email uniqueness エラー時に「Email has already been taken」が露出
- **攻撃シナリオ**: 任意の email で signup 試行 → 「Email has already been taken」を観測 = 登録済み判定
- **修正方針**: signup 失敗時はフィールド名なしの汎用文言 `{ error: "入力内容に誤りがあります..." }` 統一返却。422 ステータスは維持
- **回帰テスト**: `auth_controller_test.rb` で「email 重複時と password 短すぎ時のレスポンスボディが完全一致」を assert

### E-H2: login timing attack — 🔴 **要修正 (本 PR で fix)**

- **症状**: `auth_controller.rb:14-22` で `User.find_by` 後 `user&.authenticate` のため、email 不在時は bcrypt 計算をスキップ → 応答時間で email 存在性が漏れる
- **修正方針**: email 不在時にも **同じ cost** のダミー bcrypt を実行する。`DUMMY_DIGEST = BCrypt::Password.create("dummy", cost: BCrypt::Engine.cost).to_s.freeze` を定数化し、boot 時に 1 回のみ生成
- **回帰テスト**: 不在 email と存在 email (誤 password) のログイン応答時間差が **200ms 以内** であることを assert

### E-H3: dev seed の本番漏洩 — 🟡 **OK (docs 強化のみ)**

- **現状**: `backend/db/seeds.rb:1` `return unless Rails.env.development?` で本番ガード済 ✅
- **残リスク**: `.env.example` に `RAILS_ENV=development` がそのまま記載 → EC2 で `.env.example` をリネーム運用すると development のまま起動 → seed が動く
- **本 PR 対応**:
  - `.env.example` の該当行直上にコメント追加
  - README に本番デプロイ前チェックリスト 4 項目を追加

### E-M1: login rate limit — ❌ **未実装 (Phase 2 残課題)**

- 現状: rack-attack なし。ブルートフォース耐性ゼロ
- Phase 2: rack-attack 導入 + Issue 起票 (§5 残課題)

### E-M2: JWT blacklist — ❌ **未実装 (Phase 2 残課題)**

- 現状: `auth_controller.rb:24-27` logout は cookie 削除のみ
- Phase 2: `JwtDenylist` テーブル + jti 検証 (Devise-JWT 参考)

### E-M6: 横展開チェック (comment / like 削除の冪等性・認可) — ✅ **OK**

| エンドポイント | 確認結果 | 根拠 |
|---|---|---|
| comment delete | OK | `comments_controller.rb:17` で `@trip.comments.find(params[:id])` → 存在しない id は 404 (`rescue_from RecordNotFound`) / 他人の comment は 403 |
| like delete | OK | `likes_controller.rb:16-20` で `like&.destroy` 冪等 / 常に 200 返却 (sns-board 教訓と一致) |
| trip update/destroy | OK | `trips_controller.rb:6` で `authorize_owner!` 適用 / 他人 trip は 403 |
| me | OK (意図的) | 未ログインは `{ user: null }` の 200 返却 — ログイン状態確認専用エンドポイントとして意図的 |

### E-L1: timezone — ✅ **OK**

- `backend/config/application.rb:48-49`: `config.time_zone = "Tokyo"` / `config.active_record.default_timezone = :local` 明示済

### E-L5: signup の race condition — 🟡 **軽微 (Phase 2 残課題)**

- model validation + DB UNIQUE index あり (`create_users.rb:11`)
- ただし check→save 間で `ActiveRecord::RecordNotUnique` raise の可能性 → 現状 `application_controller.rb` の `rescue_from` 対象外のため 500 が出うる
- Phase 2: `ApplicationController` に `rescue_from ActiveRecord::RecordNotUnique` を追加し 422 で返却

### E-PWReset: パスワードリセット — ❌ **未実装 (Phase 2 着手時の Definition of Ready)**

- 現状: 未実装 (`backend/config/application.rb:10` で `action_mailer/railtie` がコメントアウト)
- **Phase 2 実装時の 5 原則を本書に固定** (sns-board followup §157-177 の流用):
  1. メール送信エンドポイントは存在 email でも不在 email でも **200 統一** (列挙防止)
  2. 同じく **レイテンシも統一** (常に enqueue まで実行 / 不在ならスキップだが sleep でフェイク)
  3. **トークンは SHA-256 ハッシュで保存** (raw 値は URL のみ / DB は hash)
  4. **消費エンドポイント** (パスワード再設定) も成功/失敗で文言・status を統一
  5. 上記 4 点それぞれに **回帰テストを書く**

### E-Proc: Issue/PR テンプレのセキュリティチェック項目化 — ❌ **本 PR で対応**

- 現状 (`.github/pull_request_template.md` 「チェックリスト」セクション): セキュリティ観点なし
- **本 PR で 3 行追加**:
  - `[ ] 認証/認可に関わる変更がある場合、docs/セキュリティ自己監査.md §2 教訓集と突合した`
  - `[ ] エラーレスポンスがフィールド名や存在性を漏らさないか確認した (E-H1)`
  - `[ ] パスワード/トークン比較で timing attack の余地がないか確認した (E-H2)`
